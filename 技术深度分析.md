# STM32F411 代码技术细节 - 深度分析

## 一、关键问题与解决方案

### 问题1: DMA资源冲突(TFT vs ADC)

**现状分析**:
```
DMA2_Stream2 已被 SPI1_TX (TFT) 占用
                                    ↓
          ADC被迫采用轮询模式 (阻塞等待)
```

**阻塞轮询的危害**:
```c
// 当前实现
while (HAL_ADC_PollForConversion(&hadc1, HAL_MAX_DELAY) == HAL_OK) {
    // 在此等待, 最长可能阻塞数秒
    // 期间无法进行其他操作 (界面卡顿!)
}
```

**完整解决方案**:

**方案A: 重新分配DMA (推荐)**
```
STM32F411 DMA2 资源:
├─ Stream0: 可用
├─ Stream1: 可用
├─ Stream2: SPI1_TX (TFT) ← 当前
├─ Stream3: ADC1 ← 迁移目标
├─ Stream4: 可用
├─ Stream5: 可用
├─ Stream6: 可用
└─ Stream7: 可用

实现步骤:
1. 修改ADC配置, 启用DMA:
   hadc1.Init.DMAContinuousRequests = ENABLE;
   
2. 在HAL_ADC_MspInit中配置DMA:
   DMA_HandleTypeDef hdma_adc;
   hdma_adc.Instance = DMA2_Stream3;
   hdma_adc.Init.Channel = DMA_CHANNEL_0;
   hdma_adc.Init.Direction = DMA_PERIPH_TO_MEMORY;
   hdma_adc.Init.Mode = DMA_CIRCULAR;  // 循环模式
   
3. 启动ADC + DMA:
   HAL_ADC_Start_DMA(&hadc1, (uint32_t*)&adc_values[0], 3);
   
4. 在DMA完成中断中处理数据:
   void HAL_ADC_ConvCpltCallback(ADC_HandleTypeDef* hadc) {
       // adc_values[] 已自动填充
       // 转换为电压值
   }
```

**方案B: ADC中断模式**
```c
// 配置ADC中断而非DMA
hadc1.Init.DMAContinuousRequests = DISABLE;
HAL_ADC_Start_IT(&hadc1);  // 启用中断模式

// 每次转换完成触发中断
void HAL_ADC_ConvCpltCallback(ADC_HandleTypeDef* hadc) {
    // 单个值已准备好
    uint16_t value = HAL_ADC_GetValue(&hadc);
    // 处理...
}
```

**性能对比**:
| 方案 | 阻塞时间 | CPU占用 | 复杂度 |
|------|---------|--------|--------|
| 轮询 (当前) | ~100μs | 10% | 低 |
| DMA | 0 | 0% | 中 |
| 中断 | 0 | 1% | 中 |

### 问题2: DHT11后台读取超时

**现象**: 
```
前台2秒读取1次: 成功率80%
后台5秒读取1次: 成功率60% (丢包)
```

**根本原因**: DHT11协议对时序的严格要求
```
关键时序参数:
├─ 初始化拉低: 18ms (允差: ±2ms)
├─ 等待应答: 20-40us (允差: ±5us)
├─ bit读取延迟: 40us (允差: ±5us)
└─ 若硬件中断延迟 > 40us → 数据错误

在STM32中:
├─ TIM11中断可能延迟数百μs
├─ 与TFT的DMA传输冲突
└─ 导致DHT11时序崩溃
```

**改进方案**:

**方案A: 在中断中禁用其他中断**
```c
void HAL_TIM_PeriodElapsedCallback(TIM_HandleTypeDef *htim) {
    if (htim->Instance == TIM11) {
        // 保存中断状态
        uint32_t primask = __get_PRIMASK();
        
        // 禁用所有中断 (包括TFT DMA完成中断)
        __disable_irq();
        
        // 在此执行DHT11读取 (关键代码段)
        DHT11_Read_Data(&temp, &humi);
        
        // 恢复中断
        __set_PRIMASK(primask);
    }
}
```

**方案B: 使用实时能力(RTOS)**
```c
// 如果项目集成了FreeRTOS:

// 创建高优先级任务
void DHT11_Task(void *pvParameters) {
    while (1) {
        vTaskSuspendAll();  // 暂停其他任务
        
        // 执行DHT11读取
        DHT11_Read_Data(&temp, &humi);
        
        xTaskResumeAll();   // 恢复任务调度
        
        vTaskDelay(pdMS_TO_TICKS(5000));  // 延迟5秒
    }
}
```

**方案C: 专用计时器 + 轮询**
```c
// 使用TIM8专门负责DHT11时序
void TIM8_Init(void) {
    // 1MHz时钟 (100MHz / 100)
    // 精确微秒计时
    
    htim8.Init.Prescaler = 99;
    htim8.Init.Period = 0xFFFF;
}

void DHT11_Delay_us_Precise(uint32_t us) {
    TIM8->CNT = 0;
    while (TIM8->CNT < us) {
        // 精确延时
        // 不受其他中断影响
    }
}
```

### 问题3: DS18B20阻塞延时

**当前代码**:
```c
float ds18b20_getTemp() {
    onewire_init();
    onewire_sendbyte(0xCC);
    onewire_sendbyte(0x44);
    
    HAL_Delay(750);  // ← 阻塞750ms! (不能响应按键!)
    
    // 读取温度...
}

// 调用时机:
if (current_menu == MENU_TEMP_HUMI) {
    ds18b20_temperature = ds18b20_getTemp();  // 整个主循环冻结750ms
}
```

**影响**:
- 按键延迟响应750ms
- UI卡顿半秒
- 菜单切换感觉迟缓

**非阻塞解决方案**:

```c
// 状态机实现非阻塞延时

typedef enum {
    DS18B20_STATE_IDLE,
    DS18B20_STATE_CONVERTING,  // 转换中
    DS18B20_STATE_READY        // 数据就绪
} DS18B20_State;

typedef struct {
    DS18B20_State state;
    uint32_t convert_start_time;
    float temperature;
} DS18B20_Context;

static DS18B20_Context ctx = {0};

// 步骤1: 启动转换 (非阻塞)
void ds18b20_startConvert(void) {
    if (ctx.state == DS18B20_STATE_IDLE) {
        onewire_init();
        onewire_sendbyte(0xCC);
        onewire_sendbyte(0x44);  // 发起转换
        
        ctx.convert_start_time = HAL_GetTick();
        ctx.state = DS18B20_STATE_CONVERTING;
    }
}

// 步骤2: 检查是否完成 (轮询调用)
uint8_t ds18b20_isReady(void) {
    if (ctx.state == DS18B20_STATE_CONVERTING) {
        // 检查是否已过750ms
        if (HAL_GetTick() - ctx.convert_start_time >= 750) {
            // 读取温度数据
            uint8_t lsb, msb;
            onewire_init();
            onewire_sendbyte(0xCC);
            onewire_sendbyte(0xBE);
            lsb = onewire_readbyte();
            msb = onewire_readbyte();
            
            // 数据转换...
            ctx.temperature = convert(lsb, msb);
            
            ctx.state = DS18B20_STATE_READY;
            return 1;  // 数据就绪
        }
    }
    return 0;  // 仍在转换中
}

// 步骤3: 获取数据
float ds18b20_getTemp(void) {
    return ctx.temperature;
}

// 使用示例:
// 在主循环中:
if (current_menu == MENU_TEMP_HUMI) {
    // 每5秒启动一次转换
    if (count % 500 == 0) {  // 假设10ms计时
        ds18b20_startConvert();
    }
    
    // 检查是否完成
    if (ds18b20_isReady()) {
        ds18b20_temperature = ds18b20_getTemp();
        // 立即更新显示 (无延迟)
    }
}
```

```c
    uint8_t dht11_humidity;      // 0-100%, 1字节足够
    int16_t ds18b20_temp_fixed;  // 0.0625°C精度的定点数
    uint16_t timestamp_sec;      // 以秒为单位, 省略毫秒

#pragma pack()

SensorData_Compact history[100];  // 600字节 (节省50%)
```

### 2. 环形缓冲区避免数据移动

```c
// 当前方案: 数据满时, 整个数组前移
if (data_index >= MAX_HISTORY) {
    for (int i = 0; i < MAX_HISTORY - 1; i++) {
        history[i] = history[i+1];  // O(n)复制
    }
}
// 代价: 每次都要移动99个数据

// 优化方案: 环形缓冲区
typedef struct {
    SensorData data[MAX_HISTORY];
    uint16_t head;   // 最新数据指针
    uint16_t count;  // 当前数据个数
} CircularBuffer;

CircularBuffer buffer = {0};

void push_data(SensorData *data) {
    buffer.data[buffer.head] = *data;
    buffer.head = (buffer.head + 1) % MAX_HISTORY;  // 循环
    
    if (buffer.count < MAX_HISTORY) {
        buffer.count++;
    }
}

// 访问第i个数据 (从新到旧)
SensorData* get_data(int index) {
    int pos = (buffer.head - index - 1 + MAX_HISTORY * 2) % MAX_HISTORY;
    return &buffer.data[pos];
}

// 代价: O(1), 无数据移动!
```

## 四、传感器协议深度解析

### 1. DHT11完整通讯波形

```
时间轴(不按比例):

主机端:
  ┌─ 初始化 (Rst)
  │ ┌─ 拉低18ms
  │ │
 HIGH ──────────────────────────┐
  │                              │
 LOW  ────────────────────────────
  │   ← 18ms ±2ms →    ← 30us →
  │
  │ ┌─ 等待应答
  │ │ ┌─ 主机拉低80us
  │ │ │
 HIGH ──────┐
  │          │ 等待从释放
 LOW  ───────┼─────────────
  │          │   80us
  │          └─→
  │
  │ ┌─ 从设备拉低80us
  │ │
 HIGH ───────────────────┐
  │                       │ 从设备拉低信号
 LOW  ────────────────────┴─────
  │                        ← 80us →
  │
  │ ┌─ 读取5字节数据 (bit位)
  │ │
  │ 对于bit='1': ┌─────────┐
  │              │ 高26~28us│
 HIGH ───────────┤          ├────
  │              └─────────┘
 LOW  ┌──────┐
      │50us  │
      └──────┘
  │
  │ 对于bit='0': ┌────────────┐
  │              │ 高70us     │
 HIGH ───────────┤            ├─
  │              └────────────┘
 LOW  ┌──────┐
      │50us  │
      └──────┘

数据格式:
  字节0: 湿度整数   (0-100)
  字节1: 湿度小数   (0-9, 当前实现未用)
  字节2: 温度整数   (0-50)
  字节3: 温度小数   (0-9, 当前实现未用)
  字节4: 校验和     = 字节0+1+2+3的低8位
```

### 2. DS18B20温度转换格式

```
温度数据 16-bit (大端序):
  ┌─ 高8位 (MSB)
  │ ┌─ 低8位 (LSB)
  
MSB LSB
 01 C8  → 0x01C8 = 456 (十进制)
           └─ 4bit整数部分 + 4bit小数部分
           
解析过程:
  整数部分 = (456 >> 4) & 0xFF = 28
  小数部分 = (456 & 0x0F) × 0.0625 = 0.5
  温度 = 28.5°C

负温度处理 (补码):
  FFFF → (-1) × 0.0625 = -0.0625°C
  FFF0 → (-16) × 0.0625 = -1.0°C
```

## 五、WiFi/MQTT通讯深度

### 1. AT指令超时处理

**当前问题**: 无超时检测
```c
// 现有实现
printf("AT+CWJAP=\"2003\",\"12121212\"\r\n");
HAL_Delay(5000);  // 固定延迟 ← 不灵活!
```

**改进方案**: 接收响应并判断

```c
#define UART_RX_BUFFER_SIZE 512
char uart_rx_buffer[UART_RX_BUFFER_SIZE];
volatile uint16_t uart_rx_count = 0;

// 中断接收回复
void HAL_UART_RxCpltCallback(UART_HandleTypeDef *huart) {
    if (huart->Instance == USART1) {
        uart_rx_count++;
        if (uart_rx_count >= UART_RX_BUFFER_SIZE) {
            // 缓冲满, 处理数据
            process_at_response();
            uart_rx_count = 0;
        }
    }
}

// AT指令执行 (带超时)
uint8_t send_at_command(const char *cmd, uint32_t timeout_ms) {
    // 清空接收缓冲
    uart_rx_count = 0;
    memset(uart_rx_buffer, 0, UART_RX_BUFFER_SIZE);
    
    // 发送命令
    printf("%s\r\n", cmd);
    
    // 等待响应
    uint32_t start = HAL_GetTick();
    while (HAL_GetTick() - start < timeout_ms) {
        // 检查是否收到 "OK"
        if (strstr(uart_rx_buffer, "OK") != NULL) {
            return 1;  // 成功
        }
        
        // 检查是否收到 "ERROR"
        if (strstr(uart_rx_buffer, "ERROR") != NULL) {
            return 0;  // 失败
        }
    }
    
    return 0;  // 超时
}

// 改进的初始化序列
void Send_AT_Commands_Robust(void) {
    if (!send_at_command("AT", 1000)) {
        printf("ESP8266 not responding!\r\n");
        return;
    }
    
    if (!send_at_command("ATE0", 500)) {
        printf("Failed to disable echo\r\n");
        return;
    }
    
    if (!send_at_command("AT+CWMODE=1", 1000)) {
        printf("Failed to set WiFi mode\r\n");
        return;
    }
    
    if (!send_at_command("AT+CWJAP=\"2003\",\"12121212\"", 10000)) {
        printf("Failed to connect WiFi\r\n");
        return;
    }
    
    printf("WiFi connected successfully!\r\n");
}
```

### 3. 内存泄漏检测

```c
// 追踪malloc/free调用 (如果使用的话)
#define MALLOC_MAX 100
typedef struct {
    void *ptr;
    uint32_t size;
    const char *file;
    int line;
} MallocRecord;

static MallocRecord malloc_records[MALLOC_MAX];
static uint16_t malloc_count = 0;

void* malloc_debug(uint32_t size, const char *file, int line) {
    void *ptr = malloc(size);
    
    malloc_records[malloc_count].ptr = ptr;
    malloc_records[malloc_count].size = size;
    malloc_records[malloc_count].file = file;
    malloc_records[malloc_count].line = line;
    malloc_count++;
    
    return ptr;
}

void malloc_report(void) {
    printf("Memory allocation report:\r\n");
    uint32_t total = 0;
    for (int i = 0; i < malloc_count; i++) {
        printf("  %s:%d: %d bytes\r\n",
               malloc_records[i].file,
               malloc_records[i].line,
               malloc_records[i].size);
        total += malloc_records[i].size;
    }
```

## 八、常见问题解答

### Q1: 为什么DHT11读取经常失败?
**A**: 通常是因为:
- 硬件中断延迟导致时序偏差
- 电源噪声干扰信号
- 数据线未正确上拉

**检查清单**:
```c
// ✓ 确认上拉电阻 (4.7k-10k)
// ✓ 确认GPIO配置正确
// ✓ 确认disable中断运行DHT11读取
// ✓ 添加校验和检测
if ((buf[0]+buf[1]+buf[2]+buf[3]) != buf[4]) {
    printf("Checksum error!\r\n");
    return 1;  // 重试
}
```

### Q2: 为什么屏幕经常闪烁?
**A**: 可能的原因:
- DMA传输与GPIO操作冲突
- SPI时钟频率过高导致丢包
- 没有正确等待DMA完成

**解决方法**:
```c
// 确保在DMA发送完成后再进行下次操作
while (HAL_SPI_GetState(ST7789_SPI) == HAL_SPI_STATE_BUSY_TX) {
    // 等待DMA完成
}

// 或使用DMA完成中断
void HAL_SPI_TxCpltCallback(SPI_HandleTypeDef *hspi) {
    dma_busy = 0;  // 清除DMA忙标志
}
```

### Q3: 为什么WiFi连接不稳定?
**A**: 常见原因:
- UART配置不正确 (波特率, 数据位等)
- ESP8266电源不稳定
- 信号干扰

**改进**:
```c

#define UART_RX_BUFFER 1024
uint8_t uart_rx_buffer[UART_RX_BUFFER];

// 使用中断+循环缓冲

    if (huart->Instance == USART1) {
        // 处理接收数据

    }
}

// 配置更稳定的时序
```
